## 1. 背景：1.0 为什么失败（复盘）

1.0（`~/Desktop/漫剧Agent`）按 24 份 PRD 实现了一个"企业级多 Agent 平台"，最终效果不可用。根因不是模型不行，而是**工程方向错了**。以下每一条都有代码证据：

| # | 根因 | 证据 | 2.0 对策 |
|---|---|---|---|
| R1 | **视频链路从未真实打通**。Seedance/Seedream/TTS/VLM 全部是确定性 Mock，输出 `mock://seedance/xxx.mp4` | `app/core/providers.py` 整个文件是 `MockProvider`；只有健康检查探针是真请求 | 设计原则 P1：第 1 周必须打通"小说→1 个真实 Seedance 视频"的垂直切片，**全项目禁止出现 mock 适配器** |
| R2 | **范围爆炸**。合规审查、预算门禁、成本台账、审计大盘、导出过期……50+ 张表、3764 行单文件服务，精力全花在流程治理 | `prd/` 下 24 份 PRD；`agents.py` 3764 行 | 2.0 P0 只做 5 件事：上传→剧本→分镜→生成→重生成。其余进非目标清单 |
| R3 | **静默兜底**。LLM 调用失败回退到模板化章节切分，垃圾数据下游传播，"看起来生成了"实际不可用 | `_fallback_extract_units` 及周边逻辑（1.0 最后一次未提交改动就是在删它） | 设计原则 P2：失败要响。任何阶段失败立即在 UI 红色可见并附原始报文，**全项目禁止静默兜底** |
| R4 | **超时与真实延迟不匹配**。代码写死 `timeout=15s`，实测 LLM 22s+、VLM 57~66s，大量假超时触发 R3 的兜底 | `urlopen(req, timeout=15)`；`store.json` 中 `provider_calls.latency_ms` 实测 21804/22759/66521ms | 超时按实测延迟的 5~8 倍设置（文本 180s、VLM 300s、视频轮询总预算 15min），见附录集成规范 |
| R5 | **单次 prompt 生成整集脚本**。一个 prompt 要求"返回严格 JSON 含 3 个方案"，无 schema 校验、无修复回路、超长小说无切分策略 | `_build_script_options_prompt` | 多阶段流水线 + 每阶段 Pydantic Schema 校验 + 带错误反馈的修复回路（详见 PROMPT_SPEC） |
| R6 | **角色/画面一致性无机制**。每个镜头 prompt 独立生成，没有角色圣经、没有参考图、镜头间无衔接 | prompt 构造代码中角色描述每次由 LLM 自由发挥 | 角色圣经 + 固定外观锚点串 + Seedream 定妆照参考图 + 场景内首尾帧衔接（§5.4） |

**一句话总结：1.0 造了一座管理大楼，但车间里没有机器。2.0 先造机器，大楼最多盖一层。**

## 2. 产品定义

### 2.1 目标

一个**单人使用**的本地创作工具：上传一本小说，产出可直接用于发布的竖屏漫剧（动态漫画短剧）视频镜头序列。核心链路：

```
上传小说 → 角色圣经 + 剧集规划 → 单集分镜脚本（画面描述/旁白/台词） → [人工确认]
        → Prompt 编译 → HiAgent Seedance 2.0 生成镜头视频 → 质检/重生成 → 成片镜头序列
```

### 2.2 成功标准（验收 DoD，"生产可用"的量化定义）

| 指标 | 目标值 | 测量方式 |
|---|---|---|
| 脚本产出速度 | 10 万字小说，≤30 分钟产出第 1 集可审分镜脚本（10~15 镜头，约 60~90s 成片时长） | 端到端计时 |
| 脚本结构合法率 | 100% 通过 Schema + 业务规则校验（时长预算、语速预算、角色引用合法） | 自动校验器 |
| 镜头一次通过率 | ≥60%（不改 prompt 直接可用） | 人工评审 20 镜头样本 |
| 重生成 1 次后可用率 | ≥85% | 同上 |
| 角色一致性 | 同一角色跨镜头人工抽检通过率 ≥80%；VLM 相似度评分 ≥0.8 | QA 阶段自动 + 人工 |
| 链路真实性 | 全链路 0 mock 数据；任意 provider 错误 UI 可见且含原始响应报文 | 代码审查 + 故障注入测试 |
| 成本可控 | 单集直接生成成本（含 40% 重生成）≤ ¥70，且有硬上限熔断 | 成本台账（轻量版） |

### 2.3 非目标（明确砍掉，写进文档防止复发）

- ❌ 多租户、权限体系、计费系统
- ❌ 独立的合规审查 Agent、版权声明流程（网关侧自带内容审核，被拒时透出报文即可）
- ❌ 预算门禁状态机（简化为"单集成本硬上限 + 超限暂停队列"一条规则）
- ❌ 审计大盘、指标告警、导出过期管理
- ❌ P0 阶段不做：TTS 配音、字幕、BGM、ffmpeg 成片合成（进 P1，见 §4.4）
- ❌ 三方案剧本对比（1.0 的"3 个 script options"没人用，2.0 一次生成一版 + 可重新生成）

### 2.4 用户与场景

唯一用户：创作者本人（你）。场景：本机运行，浏览器操作，HiAgent 网关出口。不考虑部署、扩容、多人协作。

## 3. 核心设计原则（全项目宪法，代码评审依据）

- **P1 真实链路优先**：任何阶段先用真实 API 打通最小闭环，再谈批量与优化。禁止 mock 适配器进入主干代码。
- **P2 失败要响**：模型调用失败 = 任务失败 = UI 红色 + 原始报文。禁止任何形式的静默兜底/降级模板。
- **P3 贵的环节前人工把关**：脚本阶段免费且可编辑，视频阶段烧钱（¥0.8/秒）。分镜脚本必须人工点击"确认"后才允许进入生成队列。
- **P4 一致性是系统工程**：不指望单条 prompt 写得好就一致。靠角色圣经固定锚点串（逐字复用）+ 定妆照参考图 + 首尾帧衔接三层机制（§5.4）。
- **P5 一切 LLM 输出有合同**：每个 LLM 阶段定义 Pydantic Schema + 业务规则校验器；校验失败将**具体错误**回喂模型修复，最多 2 次，仍失败则任务失败（回到 P2）。
- **P6 小而深**：一份 PRD、一个服务进程、一个 SQLite 文件、≤10 张表、≤5 个前端页面。新增任何模块需先回答"砍掉它会怎样"。

## 4. 功能需求

### 4.1 P0-1 小说摄入

| 项 | 要求 |
|---|---|
| 输入 | TXT（必须）/ EPUB（P1）。UTF-8/GBK 自动识别 |
| 处理 | 章节切分（正则识别"第X章/卷"，失败时按字数等分并提示）、广告水印行清洗、全角半角归一 |
| 长文策略 | 每章独立处理 + **滚动摘要**（rolling summary）：第 N 章处理时附带前 N-1 章的压缩摘要（≤800 字），解决跨章上下文 |
| 输出 | `chapters` 表 + 章节摘要。摄入报告：总字数、章节数、清洗行数 |
| 验收 | 100 万字小说摄入 ≤60s（纯本地处理，LLM 摘要懒加载——只在用到该章时生成） |

### 4.2 P0-2 故事策划（角色圣经 + 剧集规划）

两个独立的 LLM 阶段（prompt 见 PROMPT_SPEC §A/§B）：

**A. 角色圣经（Character Bible）**
- 输入：开头若干章 + 滚动摘要；输出：主要角色列表，每个角色含：
  - `appearance_canonical`：**固定外观锚点串**（≤60 字，发型/服装/体貌/年龄感），生成后锁定，后续所有镜头 prompt 逐字注入，禁止 LLM 改写
  - 性格、说话风格（供台词生成参考）、与其他角色关系
- 世界观：时代、题材、`visual_style_canonical`（全局画风锚点串，如"国漫厚涂风格，电影级打光，高饱和"）
- **可编辑**：UI 提供编辑界面，人工修改后锁定版本号；镜头 prompt 记录所用圣经版本

**B. 剧集规划（Episode Plan）**
- 输入：全书章节摘要；输出：剧集列表，每集含：标题、3 秒钩子、对应源章节、剧情概要、目标时长（默认 60~90s）、结尾悬念
- 漫剧节奏约束写入 prompt：前 3 秒必须有冲突/悬念钩子，每集一个情绪高点，结尾留钩

**验收**：角色圣经中主角外观锚点串在任意两次剧集生成中逐字一致；剧集规划覆盖全部选定章节、无遗漏无重叠。

### 4.3 P0-3 分镜脚本生成（核心交付物）

对单集执行，输入 = 该集源章节原文 + 剧集概要 + 角色圣经。输出每个镜头（shot）包含：

```
shot_no          镜头序号
duration_s       时长（吸附到 Seedance 合法取值，集成验证日确认，预期 4~12s）
shot_size        景别：远景/全景/中景/近景/特写
camera_move      运镜：固定/推/拉/摇/移/跟
scene_setting    场景（时间+地点+氛围，同场景镜头必须逐字复用同一场景串）
characters       出场角色（必须是圣经中存在的角色名）
action_desc      画面描述：主体+单一动作+情绪（一个镜头只一个主要动作）
narration        旁白（可空）
dialogues        台词列表 [{speaker, line, emotion}]（可空）
sfx              音效提示（P1 使用）
transition       转场：硬切/叠化/黑场
continuity       衔接标记：是否沿用上一镜头尾帧
```

**代码侧确定性校验（不依赖 LLM 自觉）**：
1. Σduration ∈ 目标时长 ±10%
2. 语速预算：每镜头（旁白+台词）字数 ≤ duration × 5 字/秒，≥ duration × 3 字/秒（除非纯画面镜头显式标注）
3. `characters` ⊆ 角色圣经；`speaker` ⊆ 该镜头 characters ∪ {旁白}
4. 单镜头 action_desc 不得包含两个以上动词短语（防止"既要又要"导致 Seedance 画面混乱）
5. 同一 `scene_setting` 的镜头必须连续编号（防止场景反复横跳）

校验失败 → 错误清单回喂 LLM 修复（最多 2 次）→ 仍失败则该集生成失败并展示错误。

**人工确认门**：脚本以表格形式在 UI 呈现，逐镜头可编辑（编辑后重过校验器）。点击"确认分镜"后才解锁生成按钮（原则 P3）。

**验收**：连续生成 5 集，结构合法率 100%；人工通读脚本，旁白+台词与原著情节一致、无角色张冠李戴。

### 4.4 P0-4 Prompt 编译（确定性为主）

**这是 1.0 完全缺失的一层。** 分镜脚本 ≠ Seedance prompt。编译器是**确定性代码**（非 LLM），模板：

```
[visual_style_canonical] ‖ [景别+运镜的镜头语言] ‖ [scene_setting]
‖ [每个出场角色的 appearance_canonical 逐字注入] ‖ [action_desc]
‖ (若 Seedance 2.0 支持原声台词：台词文本)
--ratio 9:16 --dur {duration} [--resolution 1080p] [其余参数见集成规范]
```

编译规则：
- 长度校验：≤ Seedance prompt 字符上限（集成验证日实测，1.0 记录为 1800），超长时按"台词>动作>场景>风格"的优先级保留，截断角色锚点串以外的修饰语
- 负向词表：统一维护一份（变形手指/多余人物/文字水印/真人明星脸等），不在每镜头重复生成
- 参考图注入：该镜头出场角色的定妆照（§5.4）作为 reference image；`continuity=true` 的镜头将上一镜头尾帧作为 first_frame
- 每次编译产出带版本号的 prompt 记录，人工可在生成前/重生成时直接改 prompt 文本（改完仍过长度校验）

**验收**：100 个镜头编译 0 次 API 400 错误（参数校验全部前置）。

### 4.5 P0-5 视频生成（HiAgent Seedance 2.0）

详细 API 形态见 [docs/HIAGENT_INTEGRATION.md](docs/HIAGENT_INTEGRATION.md)。要求：

- **异步任务模式**：创建任务 → 轮询状态 → 下载视频到项目目录 `projects/<id>/episodes/<n>/shots/<m>/v<k>.mp4`
- **本地队列**：并发上限可配（默认 2，1.0 配置经验值），FIFO + 失败重试 2 次（指数退避），429 自动降速
- **幂等**：以 `hash(prompt + 参数 + 参考图)` 为幂等键，相同请求复用已有结果（重生成时改了 prompt 自然产生新键）
- **成本熔断**：每集累计成本 = Σ(时长 × ¥0.8) 实时累加，超过单集上限（默认 ¥100，可配）暂停队列并提示
- **生成历史**：每个镜头保留全部版本（prompt + 视频 + 任务报文 + 耗时 + 成本），UI 可回滚选用任意历史版本

**验收**：批量提交 15 镜头，全部成功落盘或明确失败原因可见；中途杀进程重启后队列状态可恢复（任务表持久化）。

### 4.6 P0-6 质检与重生成

- **自动 QA（VLM）**：每个镜头生成后用 VLM 模型（chat/completions 传视频帧或视频 URL，能力以集成验证日为准；不支持视频则抽 3 关键帧）检查：① 角色外观与锚点串相符 ② 画面与 action_desc 相符 ③ 无文字/水印/多余人物。输出 0~1 分 + 问题描述
- **自动重试**：QA <0.6 且重试次数 <1 时，将 QA 问题描述拼入 prompt 负向区自动重生成一次
- **人工评审墙**：镜头卡片流（视频 + prompt + QA 分数并排），操作：✅采用 / ✏️改 prompt 重生成 / 🔁原 prompt 重抽（换 seed）
- QA 不阻塞流程：QA 失败（模型超时等）只标记"未质检"，人工照常可看可采用（P2：失败要响但不卡死）

**验收**：评审墙上从"看到问题"到"提交重生成"≤3 次点击。

### 4.7 P1（明确排期但不在 P0 交付）

1. **TTS 配音**：旁白/台词走 HiAgent TTS（或 Seedance 2.0 原声能力，集成验证日确认），按 emotion 选音色
2. **成片合成**:ffmpeg 按 shot_no 拼接 + 台词字幕烧录（数据已在脚本中，纯工程）
3. **EPUB 摄入**、批量整本生成（逐集排队过夜跑）
4. **Seedream 定妆照工作流 UI 化**（P0 阶段定妆照生成走脚本/接口即可，见 §5.4）

## 5. 系统设计

### 5.1 技术栈（刻意保守）

| 层 | 选型 | 理由 |
|---|---|---|
| 后端 | Python 3.12 + FastAPI + Uvicorn | 沿用 1.0 技术认知，零学习成本 |
| 存储 | SQLite（SQLModel/SQLAlchemy）+ 文件系统存媒体 | 单用户本地工具，杀掉 1.0 的 JSON 大文件存储（已出现读写竞争隐患） |
| 任务 | asyncio 后台队列（进程内），任务表持久化保证重启恢复 | 不引入 Celery/Redis |
| 前端 | React + Vite（复用 1.0 技术栈）但只有 **5 个页面** | 项目页 / 圣经&剧集页 / 分镜工作台 / 生成监控 / 评审墙 |
| LLM SDK | OpenAI 兼容客户端（httpx），统一指向 HiAgent 网关 | 网关即 OpenAI 协议（1.0 已验证） |

### 5.2 数据模型（≤10 张表）

```
projects        项目（小说名、状态、成本累计、圣经版本指针）
chapters        章节（原文、摘要、清洗报告）
characters      角色圣经（appearance_canonical、版本号、锁定标记）
episodes        剧集（规划信息、状态：planned/scripted/confirmed/generating/done）
shots           镜头（§4.3 全部字段 + 当前采用版本指针）
shot_versions   镜头版本（prompt、参数、视频路径、QA 结果、成本、任务报文）
jobs            生成任务队列（持久化，重启恢复）
provider_calls  全部对外调用日志（请求/响应摘要、延迟、状态码）——轻量审计，一张表足够
settings        全局配置（网关地址、模型 ID、并发、成本上限）
```

### 5.3 目录即资产

```
projects/<project_id>/
├── novel.txt                     原文
├── bible.json                    圣经快照（每版本一份）
├── refs/<character>.png          定妆照
└── episodes/<n>/
    ├── script.json               分镜脚本（确认版快照）
    └── shots/<m>/v<k>.mp4        镜头视频各版本
```

媒体永远在文件系统，数据库只存路径。删项目 = 删目录 + 级联删行，无对象存储依赖。

### 5.4 一致性三层机制（对应 R6；2026-06-12 已全部实测落地）

1. **锚点串逐字注入**：角色外观/画风/场景串生成一次、人工校订、锁定，编译器逐字拼接，LLM 永不复述改写（改写=漂移）
2. **Seedream 定妆照**：圣经定稿后，对每个角色按锚点串生成全身立绘（1440x2560，与视频同比例）存入 `refs/`；以 base64 data URL 作为 `reference_image` 传给 Seedance。实测角色发型/服装/五官与定妆照保持一致（样片 `m0_samples/ref_test.mp4`）
3. **首尾帧链式衔接**：`continuity_from_prev=true` 的镜头与上一镜头构成依赖链——等上一镜成片后本地 ffmpeg 抽尾帧（网关不回传 last_frame_url），以 data URL 作为 `first_frame` 输入下一镜

**实测约束（决定机制拓扑）**：`first_frame` 与 `reference_image` 不能同时使用（网关 400）。因此：**链头镜头带定妆照锚定角色身份，链中镜头只带首帧**——角色一致性由链头注入，沿帧序传递；分镜阶段铁律要求同场景内除首镜外全部成链。

### 5.5 可观测性（轻量但必须）

- 每次对外调用写 `provider_calls`：端点、模型、延迟、状态码、错误报文前 500 字
- 生成监控页：队列长度、运行中任务、单集成本实时累计、最近 10 条失败及报文
- 不做：指标聚合、告警、大盘（非目标）
